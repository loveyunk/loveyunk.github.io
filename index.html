docker常用命令

发表于 2018-07-29 | 分类于 Docker |
  • docker –help
  • docker image –help
  • docker container –help

  • docker images = docker image ls 查看已有的docker镜像
  • docker ps = docker container ls 查看当前正在运行的容器
  • docker ps -a = docker continer ls -a 查看所有容器列表(包括正在运行的和已经被停止的)
  • docker start/stop id/name 启动/停止某个容器
  • docker rm id/name 删除某个容器
  • docker rmi id/name 删除某个镜像
  • docker attach id 进入某个容器(使用exit退出后容器也跟着停止运行)

docker exec与docker attach区别

  • docker attach id
  • exit
    执行以上命令可以看到attach可以进入到一个已经运行的容器，进行命令操作，但是从容器中退出之后容器也停止了，所以在生产环境中很少使用
  • docker exec -it id bash
  • exit
    使用docker exec -it进入容器和我们平常操作console界面类似。而且不像attach方式退出，导致整个容器退出，生产环境中用的比较多。比使用ssh 、nsenter、nsinit方式更方便，生产中常用的方式

docker container run xxx 与 docker start xxx

  • docker container run 命令会从 image 文件，生成一个正在运行的容器实例。注意，docker container run命令具有自动抓取 image 文件的功能。如果发现本地没有指定的 image 文件，就会从仓库自动抓取。
  • 每运行一次 docker container run xxx 都会重新生成一个容器。
  • docker start 重启一个已经生成的容器。

export default

  • 一般用在一个文件就是一整个模块，将整个模块导出
  • 单文件模块
  • 无需命名，将命名权交给import
  • import时不需要大括号

export default 与 export

  • export deafult是集中的，export是分散的
  • import的时候，一个需要大括号，一个不需要大括号
  • 单文件模块用export deafult,多模块用export
  • export {a, b}导入需要import {a, b} from 'xx'
  • export default {a, b}导入需要import foo from 'xx'使用则为foo.x

export default {a: 1, b: 2} 导出

  • import foo from 'XX'; 则使用时可以 foo.a, foo.b
  • import foo from 'XX'; 则使用时可以 const {a, b} = foo 用解构赋值先解构出来。

nginx配置文件详解


nginx的location配置详解

  • 语法规则：location [=|~|~*|^~] /uri/ { … }
  • = 开头表示精确匹配
  • ^~ 开头表示uri以某个常规字符串开头，理解为匹配url路径即可。nginx不对url做编码，因此请求为/static/20%/aa，可以被规则^~ /static/ /aa匹配到（注意是空格）。
  • ~ 开头表示区分大小写的正则匹配
  • ~* 开头表示不区分大小写的正则匹配
  • !~和!~*分别为区分大小写不匹配及不区分大小写不匹配 的正则
  • / 通用匹配，任何请求都会匹配到。
  • 多个location配置的情况下匹配顺序为：首先匹配 =，其次匹配^~, 其次是按文件中顺序的正则匹配，最后是交给 /通用匹配。当有匹配成功时候，停止匹配，按当前匹配规则处理请求。

如何生成ssh key

发表于 2018-07-29 | 分类于 Git |

SSH key是建立在应用层和传输层基础上的安全协议。提供了一种与GitHub通信的方式，通过这种方式，方便访问github，
本地生成ssh公钥，添加到github上，这样就不用每一次访问github都要输入用户名和密码。

步骤

  • 检查SSH keys是否存在
  • 生成新的ssh key
  • 将ssh key添加到GitHub中

1. 检查SSH keys是否存在

  • 在创建SSH keys之前，我们可以看看电脑内是否有SSH key。
  • 输入ls -al ~/.ssh
  • 如果看见如下文件，则代表SSH key已创建好。

    id_dsa.pub

    id_ecdsa.pub

    id_ed25519.pub

    id_rsa.pub

2. 生成新的ssh key

  1. 设置电子邮件并创建对应的key
    ssh-keygen -t rsa -C "email@email.com"
  2. 设置文件存储位置，直接“回车”。

    Enter a file in which to save the key (/Users/you/.ssh/id_rsa): [Press enter]
  3. 设置密码时，可设置空密码。

3. 将ssh key添加到GitHub中

Git团队协作

发表于 2018-07-27 | 分类于 Git |

习惯养成

如果一个团队在使用 Git 时没有一些规范，那么将是一场难以醒来的噩梦！然而，规范固然重要，但更重要的是个人素质，在使用 Git 时需要自己养成良好的习惯。

分支管理

get不同分支的职责

  • master – 即主分支。任何项目都必须有个这个分支。对项目进行tag或发布版本等操作，都必须在该分支上进行。
  • dev – 即开发分支，从master分支上检出。团队成员一般不会直接更改该分支，而是分别从该分支检出自己的个人分支，开发完成后将feature分支上的改动merge回dev分支。
  • release – 版本发布分支
  • feature – 功能分支
  • bug – bug分支

操作流程

  1. 首先在github建立项目master分支（添加README），然后由master分支建立dev分支
  2. 项目负责人克隆master分支，然后切换dev分支且与远程分支关联 git checkout -b dev origin/dev
  3. 项目成员只需克隆dev分支 git clone -b dev url
  4. 然后由dev分支建立自己的个人本地分支 git checkout -b xxx，日常开发在xxx分支上进行，commit但不push
  5. 合并xxx分支到dev分支，首先切换到dev分支，然后 git merge –no-ff -m “message” xxx
  6. 将本地dev分支push进远程dev分支
  7. 其他成员在dev分支 git pull 拉取远程分支，然后将dev分支合并到xxx分支 git merge –no-ff -m “message” dev

vue-cli打包生成文件

  • app.js，是我们项目的公用JS代码，入口就是main.js，这个公用代码，就是指除了在node_moudles公用包之外的公用代码。

  • vendor.js是从node_modules包中提取的公用的JS代码库，与app.js不同，这个里面无业务JS。
